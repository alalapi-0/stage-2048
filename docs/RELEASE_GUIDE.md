# /docs/RELEASE_GUIDE.md
# 说明：指导如何发布 Stage2048 的 Web 与小程序版本，并维护版本记录。

## Web 发行

### 方案 A：静态资源托管
1. 在服务器或任意静态空间中上传 `/web/` 目录下的全部文件。
2. 确保服务端返回 `index.html`，浏览器即可直接运行，不需要后端依赖。
3. 若需自定义默认参数，可在部署链接后添加查询参数（例：`?size=3&gap=8`）。

### 方案 B：GitHub Pages
1. 在仓库设置页打开 `Pages` 功能，选择 `main` 分支并指定 `/web` 子目录。
2. 部署完成后访问 GitHub Pages 域名，确认 `Stage2048` 可正常加载。
3. 如需自定义域名，按 GitHub Pages 文档添加 `CNAME` 并验证 DNS。

## 小程序发布

1. 在 `miniprogram/project.config.json` 中替换为自己的 `appid`，保持 `miniprogramRoot` 指向源代码目录。
2. 使用微信开发者工具导入项目，确保通过基本代码检查后再提交审核。
3. 在真机预览阶段测试存储、画布与触摸交互，确保与 Web 版本体验一致。
4. 提交审核时确认应用不收集个人信息、不请求额外权限，以满足合规要求。

## 版本号建议

1. 在根 `README.md` 顶部记录当前版本号，例如从 `v0.1.0` 提升至 `v1.0.0` 时同步更新徽章或文本。
2. 每次发布前梳理新增特性、修复与破坏性变更，确保版本号遵循语义化规则。
3. 对应的发布说明可在 GitHub Releases 中粘贴自 `README.diff.md` 的摘要，保持信息一致。

## 变更记录维护

1. `README.diff.md` 用于记录每一轮的重要改动与验证步骤，请在合并前更新。
2. GitHub Releases 或外部公告可引用 `README.diff.md` 的摘要，保持用户可追溯性。
3. 若未来接入排行榜或云存储等功能，建议在此文档增加专门章节描述额外的上线检查项。
